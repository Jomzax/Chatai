# Architecture Decisions

## Decision 1: เลือกใช้ MongoDB เป็น database สำหรับ Chat AI

### Context
โปรเจกต์ Chat AI นี้มีข้อมูลหลายแบบที่เปลี่ยนโครงสร้างได้บ่อย เช่น conversation history, รายการข้อความผู้ใช้/AI, ของไฟล์ที่อัปโหลด และ ผลเอกสาร ข้อมูลเหล่านี้ไม่ได้เป็น relational แบบตารางที่ตายตัวตั้งแต่แรก เพราะระหว่างพัฒนาต้องเพิ่ม field ใหม่ค่อนข้างบ่อยเพื่อรองรับฟีเจอร์ เช่น token usage, citation, extraction status และข้อความอ้างอิงจากไฟล์ PDF/TXT

### Alternatives Considered
SQL (PostgreSQL/MySQL): เหมาะกับข้อมูลที่มีโครงสร้างชัดเจนและต้องการความสัมพันธ์ระหว่างตาราง แต่จะต้องทำ migration บ่อยเมื่อโครงสร้างข้อมูลเปลี่ยนบ่อย และอาจทำให้การพัฒนาในช่วงแรกช้าลง เพราะต้องวางแผน schema ให้ดีกว่านี้เพราะเวลา 5 วันเป็นขีดจำกัด

### Why MongoDB
ที่ผมเลือก MongoDB เพราะตอนทำโปรเจกต์นี้ requirement เปลี่ยนค่อนข้างบ่อย โดยเฉพาะฝั่ง chat, history และข้อมูลไฟล์ที่อัปโหลด การใช้ document model ทำให้โครงสร้างข้อมูลยืดหยุ่นกว่าและปรับ field ใหม่ได้เร็ว ไม่ต้องเสียเวลาทำ migration หนักทุกครั้งเหมือนตอนใช้ schema ที่ตึงมาก อีกอย่างหนึ่งคือข้อมูลของเราเป็นลักษณะ "ข้อความ + metadata" ซึ่งเข้ากับรูปแบบ document โดยตรง ทำให้ทั้งอ่านและเขียนโค้ดง่ายขึ้น และช่วยให้โฟกัสกับการส่งฟีเจอร์ได้ก่อน สุดท้าย MongoDB ยังเข้ากับ ที่ใช้ (Node.js/Express) และต่อกับ session ได้ลื่น จัดการ auth/session  ได้ค่อนข้างตรงไปตรงมาในภาพรวมของระบบตอนนี้ถ้าอยากให้เติบโตต่อไปได้ง่ายๆ โดยไม่ต้องมานั่งแก้ schema บ่อยๆ

### Trade-offs
สิ่งที่ต้องยอมรับคือ พอไม่ใช้ relational database ก็จะเสียความสามารถในการทำ query ที่ซับซ้อนหรือการ join ข้อมูลข้าม collection ได้ง่ายๆ ซึ่งอาจจะต้องเขียนโค้ดเพิ่มขึ้นในบางกรณีเพื่อดึงข้อมูลที่สัมพันธ์กันมาใช้ แต่สำหรับโปรเจกต์นี้ที่เน้นความเร็วในการพัฒนาและความยืดหยุ่นของข้อมูลเป็นหลัก ผมคิดว่า MongoDB  เป็นตัวเลือกที่เหมาะสมที่สุดในตอนนี้

## Decision 2: เลือกใช้ Express.js เป็น Backend Framework

### Context
โปรเจกต์นี้เป็นงานการพัฒนาเว็บแอปแชท AI ที่ต้องส่งภายใน 5 วัน โดยโจทย์กำหนดฟีเจอร์หลักครบทั้ง Login + Protected Routes, Chat กับ AI, Upload ไฟล์ (PDF/TXT) และ Token Usage พร้อมเงื่อนไขให้เลือกได้ว่าจะใช้ API Routes ของ Next.js/Nuxt.js หรือแยก backend ออกมาต่างหาก ดังนั้นก่อนเริ่มลงมือจริง ผมจึงต้องตัดสินใจโครงสร้าง backend ที่เหมาะกับเวลาและขอบเขตงานมากที่สุด โดยเน้นให้พัฒนาได้เร็ว แก้ไขง่าย ทดสอบได้ไว และรองรับการขยายฟีเจอร์ต่อได้โดยไม่ต้องรื้อระบบกลางคัน ซึ่งถ้าเลือกใช้ API Routes ใน Next.js/Nuxt.js ก็จะได้ความง่ายในการตั้งค่าและ deployment ที่รวมทุกอย่างไว้ที่เดียว แต่ก็อาจจะเจอข้อจำกัดเรื่องการจัดการ session, auth, rate limit และการเชื่อมต่อกับ AI provider ที่ซับซ้อนขึ้นเมื่อระบบใหญ่ขึ้น ในขณะที่ถ้าแยก backend ออกมาต่างหาก เราจะมีความยืดหยุ่นในการเลือกเทคโนโลยีและออกแบบระบบได้ตรงกับความต้องการมากขึ้น โดยเฉพาะเรื่องการจัดการ session และ auth ที่เป็นหัวใจสำคัญของระบบนี้ และยังช่วยให้การพัฒนาและทดสอบ backend เป็นอิสระจาก frontend ทำให้สามารถโฟกัสกับการพัฒนาฟีเจอร์ได้เต็มที่โดยไม่ต้องกังวลเรื่องการเชื่อมต่อระหว่าง frontend กับ backend ในช่วงแรก

### Alternatives Considered
ตามโจทย์จริงสามารถทำได้ 2 แนวทางหลัก: 1. ใช้ API Routesใน Next.js/Nuxt.js หรือ 2. แยก backend ต่างหากได้ สุดท้ายเลือกแบบแยก backend เพราะควบคุม auth/session, rate limit, file upload และการเชื่อมต่อ AI provider ได้ชัดกว่า  เมื่อเลือกแยก backend แล้ว

### Why Express.js
ที่ผมเลือก Express.js เป็น backend framework เพราะมันเป็น framework ที่เบาและยืดหยุ่นมากที่สุดใน โปรเจคนี้ ซึ่งเหมาะกับโปรเจกต์ที่ต้องการความเร็วในการพัฒนาและไม่ต้องการโครงสร้างที่ตายตัวตั้งแต่แรก เพราะเวลามีจำกัดแค่ 5 วัน การใช้ framework จะช่วยให้เราสามารถเริ่มเขียนโค้ดได้เร็วขึ้น และสามารถหาตัวอย่างหรือแก้ปัญหาที่เจอได้ง่ายกว่า framework ที่ซับซ้อนกว่าและมีประสบการณ์ใช้งานอยู่แล้วไม่ต้องเรียนรู้ใหม่เยอะทำให้เราสามารถโฟกัสกับการพัฒนาฟีเจอร์ได้เลย

### Trade-offs
ข้อเสียของการเลือก Express.js ทำให้เราได้ความเร็วและความยืดหยุ่นสูงในช่วงพัฒนา แต่สิ่งที่ต้องแลกคือ framework นี้ไม่บังคับโครงสร้างโปรเจกต์แบบเข้มเหมือน NestJS ดังนั้นต้องคุมมาตรฐานเองมากขึ้น เช่น การแยก controller/service, การจัดการ error handling, การตั้งชื่อไฟล์ และการดูแล middleware ให้สม่ำเสมอ อีกจุดคือเมื่อระบบใหญ่ขึ้น หากไม่มี convention ที่ชัดเจน โค้ดอาจกระจายและดูแลยากขึ้นได้ นอกจากนี้ Express ไม่ได้มี บางอย่างมาให้สำเร็จรูปเหมือน framework ตัวอื่น ทำให้ต้องเขียนเอง อย่างไรก็ตาม สำหรับโปรเจกต์นี้ที่มีเวลา 5 วันและต้องส่งฟีเจอร์หลักให้ครบ ผมมองว่านี้ก็ยังยอมรับได้ และคุ้มกับความเร็วในการพัฒนาที่ได้กลับมาและไม่จำเป็นต้องเรียนรู้ framework ใหม่ที่ซับซ้อนกว่าในเวลาจำกัดนี้ครับ

## Decision 3: การเลือกใช้ Multi-provider AI (Gemini เป็นหลัก, Anthropic เป็น fallback)

### Context
ระบบแชทของโปรเจกต์นี้พึ่งพา AI API จากผู้ให้บริการภายนอกโดยตรง จึงมีความเสี่ยงด้านความพร้อมใช้งาน เช่น quota/rate limit, billing/credit และปัญหาทางเทคนิคของ provider หากใช้ provider เดียว เมื่อเกิดเหตุขัดข้องผู้ใช้อาจไม่สามารถแชทต่อได้ทันที ดังนั้นระบบจึงต้องรองรับการทำงานแบบ multi-provider โดยใช้ Gemini เป็น provider หลัก และใช้ Anthropic เป็น fallback เพื่อให้บริการต่อเนื่องได้แม้ provider หลักมีปัญหาชั่วคราว

### Alternatives Considered
- ใช้ Gemini เพียง provider เดียว: ตั้งค่าง่ายและดูแลง่าย แต่มีความเสี่ยงสูงถ้า โคตา หมดหรือบริการมีปัญหา เพราะระบบจะไม่มีทางสำรอง
- ใช้ Anthropic เพียง provider เดียว: ลดความซับซ้อนเช่นกัน แต่ยังคงมีปัญหา โคตาการใช้งาน แบบเดียวกัน

### Why Multi-provider AI
ที่เลือกใช้ Multi-provider AI เพราะช่วยเพิ่มความต่อเนื่องของระบบแชทได้ชัดเจนที่สุดในสถานการณ์ใช้งานจริง โดยกำหนดให้ Gemini เป็น provider หลักสำหรับการตอบคำถามปกติ และใช้ Anthropic เป็น fallback เมื่อคำขอเริ่มต้นไม่สามารถทำงานได้ เช่น เจอ quota limit, billing/credit issue หรือปัญหาชั่วคราวของ provider หลัก แนวทางนี้ช่วยลดโอกาสที่ผู้ใช้จะเห็นข้อความล้มเหลวและต้องลองใหม่หลายรอบ นอกจากนี้ยังทำให้ทีมควบคุมความเสี่ยงได้ดีขึ้น เพราะไม่ผูกระบบไว้กับผู้ให้บริการรายเดียว เมื่อมีเหตุขัดข้องเรายังรักษาเสถียรภาพของประสบการณ์ผู้ใช้ได้ และสามารถวางนโยบายการสลับ provider ให้เหมาะกับต้นทุนและคุณภาพคำตอบในแต่ละช่วงเวลา และนอกจากนี้มันเป็ยการออกแบบที่ยืดหยุ่นในระยะยาว

### Trade-offs
การใช้ Multi-provider AI ช่วยเรื่องความเสถียร แต่ต้องแลกกับความซับซ้อนในการดูแลระบบมากขึ้นอย่างชัดเจน ต้องจัดการคีย์หลายชุด, ตั้งค่า environment ให้ถูกทุกตัว, และออกแบบ fallback logic ให้ปลอดภัยไม่สลับ provider ผิดจังหวะ นอกจากนี้พฤติกรรมคำตอบของแต่ละ provider อาจต่างกันทั้งสไตล์ภาษา ระดับความยาว และรูปแบบการสรุป ทำให้ต้องมีกติกากลางเพื่อคุมคุณภาพให้ใกล้เคียงกัน อีกด้านคือการ monitor และ debug จะยากขึ้น เพราะต้องแยกให้ออกว่าปัญหาเกิดจาก provider ไหนและเกิดในขั้นตอนไหน สุดท้ายต้นทุนโดยรวมอาจคาดการณ์ยากขึ้นหากมีการสลับไป fallback บ่อย จึงต้องมีการติดตาม usage และปรับนโยบาย rate limit/rotation ให้อย่างสม่ำเสมอกับโปรเจคนี้
